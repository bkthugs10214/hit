## Decisions

- **Mermaid for all diagrams** — see [ADR 0001](../decisions/0001-mermaid-for-diagrams.md). Renders natively on GitHub, lives in markdown alongside code, diffs cleanly in PRs. Alternatives (PlantUML, draw.io, Excalidraw) each fail at least one of those properties.
- **Three-phase split (Deploy / Runtime forecast / Evaluation)** — matches the natural lifecycle and the boundaries an engineer would actually look at separately. Other groupings considered: per-module, per-asset, per-data-source. Rejected because they don't carry the temporal/operational meaning that "phase" implies.
- **One workflow + one component diagram per phase** — workflow shows *what happens in what order*, component shows *what talks to what*. They answer different questions; an engineer reading code needs both.

## Problems & resolutions

- **Problem:** "Phases" was ambiguous when the user asked for diagrams — could have meant pipeline steps inside `forward()`, the README roadmap items, or system lifecycle stages.
  **Resolution:** Confirmed with user before writing diagrams. Lifecycle interpretation chosen.
- **Problem:** Single combined runtime diagram risked being too dense for slide use.
  **Resolution:** Split into two diagrams (sequence for workflow, graph for component). The sequence diagram uses shaded `rect` blocks to separate primary vs. fallback paths so they can be presented one at a time.

## Back-and-forth / rejected approaches

- Considered ASCII art instead of Mermaid for terminal-only viewing; rejected because Mermaid's GitHub rendering is more presentation-friendly. ASCII can be added later if a CI/CD log diagram is needed.
- Considered putting all diagrams in a single `ARCHITECTURE.md`; rejected because phase-per-file means engineers can grep/link directly to the phase they're touching without scrolling past unrelated content.
- Considered backfilling ADRs for choices already baked into the code (e.g. "why JSONL not SQLite" — gap #8 in memory); rejected per the journal skill rule "don't backfill dates / don't fabricate yesterday's entry." Those choices are documented well enough by the project memory and `git log`. New ADRs from this point forward only.
